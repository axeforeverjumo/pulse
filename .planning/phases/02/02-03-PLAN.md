Per EXEC-04: Add validation in `add_agent_assignee_endpoint()` (line 2587) that rejects assigning OpenClaw agents (tier "core" or "advance"/default) to dev tasks.

After `assignee = await add_agent_assignee(...)` at line 2596 succeeds (this creates the DB row), we need to validate BEFORE the enqueue step. Actually, it is better to validate BEFORE the assignment to avoid creating a row we immediately reject.

Insert validation BEFORE `assignee = await add_agent_assignee(...)`:

```python
# EXEC-04: Reject OpenClaw agents on dev tasks
from lib.supabase_client import get_async_service_role_client
supabase_for_validation = await get_async_service_role_client()

# Fetch agent tier
agent_check = await supabase_for_validation.table("openclaw_agents")\
    .select("id, tier")\
    .eq("id", request.agent_id)\
    .maybe_single()\
    .execute()
agent_tier = (agent_check.data or {}).get("tier", "advance") if agent_check else "advance"

# Fetch issue + board to resolve dev task status
issue_for_check = await supabase_for_validation.table("project_issues")\
    .select("id, board_id, is_dev_task")\
    .eq("id", issue_id)\
    .maybe_single()\
    .execute()
if issue_for_check and issue_for_check.data:
    board_for_check = await supabase_for_validation.table("project_boards")\
        .select("id, is_development")\
        .eq("id", issue_for_check.data.get("board_id"))\
        .maybe_single()\
        .execute()
    is_dev = _resolve_effective_is_dev_task(
        issue_for_check.data,
        board_for_check.data or {},
    )
    if is_dev and agent_tier != "claude_code":
        raise HTTPException(
            status_code=status.HTTP_400_BAD_REQUEST,
            detail="Las tareas de desarrollo solo aceptan el agente Claude Code. Los agentes OpenClaw no pueden asignarse a tareas de desarrollo.",
        )
```

NOTE: `supabase_for_validation` uses the service role client because `add_agent_assignee` uses the user JWT. We need service role to query openclaw_agents without RLS issues. The import for `get_async_service_role_client` is already used elsewhere in the file.

This validation runs BEFORE the actual assignment. If the check fails, the agent is never assigned and the user gets a clear 400 error.

The reverse direction (assigning Claude Code to a non-dev task) is allowed -- no restriction needed.
